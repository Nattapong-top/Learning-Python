🧠 ความลับของโจทย์ยาก (ที่ dev เก่งรู้กัน)

โจทย์ยาก ไม่ใช่ logic ยาวขึ้น
แต่คือ:

มีหลาย “โจทย์ย่อย” ซ่อนอยู่ในโจทย์เดียว

ถ้าไม่แตก → สมองตัน
ถ้าแตกเป็นชิ้น → ง่ายลงทันที

🔓 วิธีรับมือโจทย์ยาก (สูตรรอดจริง)
🥇 กฎเหล็กข้อเดียว

ห้ามแก้โจทย์ยากตรง ๆ

ต้องทำแบบนี้แทน 👇

🧩 สูตร 4 ขั้น ตอนเจอโจทย์โหด
① “แยกโจทย์” (Decomposition)

ถามตัวเอง:

ต้องอ่านข้อมูลยังไง

ต้องจัดรูปข้อมูลไหม

ต้องคำนวณอะไร

ต้องสรุปผลอะไร

📌 ถ้าแยกได้ → รอด

② “ทำโจทย์ย่อยให้โง่ที่สุด”

โจทย์ย่อยทุกข้อ ต้องง่ายจนตอบได้ทันที

เช่น:

หาค่ามากสุดใน list

นับจำนวนตัวที่เท่ากัน

ข้ามค่าที่ไม่ใช่ int

👉 ป๋า ทำผ่านแล้วทั้งหมด 😏

③ “ต่อชิ้นส่วน”

เอา logic ย่อยมาเรียงลำดับ:

อ่าน → กรอง → คำนวณ → สรุป

ไม่ต้องคิดทีเดียวทั้งก้อน

④ “ยอมให้ logic ยาว”

อันนี้สำคัญมาก ⚠️

logic ที่ยาว ≠ คิดไม่เป็น
logic ที่ยาว = คิดครบ

dev senior ชอบ logic ยาว แต่ไม่มั่ว

🧠 เปรียบเทียบให้เห็นภาพ
🧩 โจทย์ง่าย (ที่ป๋าทำ)

หาเลขมากสุด
= 1 logic chunk

🧩 โจทย์ยาก

อ่านหลายบรรทัด → แยกข้อมูล → กรอง → คำนวณ → เปรียบเทียบ → สรุป
= 5–6 logic chunk

แต่ละ chunk
👉 คือโจทย์ easy ที่ป๋าทำผ่านมาแล้ว

🎯 Mindset ที่ต้องเปลี่ยน (สำคัญมาก)

เปลี่ยนจาก:

“โจทย์นี้ยากจัง”

เป็น:

“โจทย์นี้มีกี่ชิ้น?”

🔥 ประโยคทอง (จำให้ขึ้นใจ)

“โจทย์ยาก ไม่ได้ยากเพราะมันโหด
แต่มันยากเพราะยังไม่ได้แตก”
